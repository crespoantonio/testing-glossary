# Glosario de Testing y QA

## A

- **A/B Testing:** Comparación entre dos versiones para evaluar cuál rinde mejor.
- **Accessibility Testing:** Verifica la usabilidad por personas con discapacidades.
- **Acceptance Criteria (Criterios de Aceptación):** Condiciones específicas que una funcionalidad debe cumplir para ser aceptada.
- **Acceptance Testing (Pruebas de Aceptación):** Verifican si el sistema cumple con los requisitos del cliente o usuario final.
- **Ad Hoc Testing:** Pruebas informales sin planificación para encontrar errores de forma intuitiva.
- **Agile Testing (Pruebas Ágiles):** Pruebas continuas integradas en el desarrollo ágil.
- **Alpha Testing:** Pruebas internas hechas por el equipo antes de liberar el producto.
- **API Testing:** Pruebas directamente a las interfaces de programación de aplicaciones (APIs), a menudo utilizando herramientas como Postman o SoapUI
- **Automated Testing (Pruebas Automatizadas):** Ejecución de pruebas con herramientas automáticas.

## B

- **Behavior Driven Development (BDD):** Metodología que enfatiza la colaboración mediante lenguaje natural para definir comportamientos del sistema.
- **Beta Testing:** Pruebas realizadas por usuarios reales en entornos reales.
- **Black Box Testing:** Pruebas sin conocimiento del código interno. Solo se validan entradas y salidas.
- **Boundary Value Analysis:** Técnica que prueba los límites de entrada válidos e inválidos.
- **Bug:** Defecto en el software que causa un comportamiento incorrecto.
- **Bug Tracking System:** Herramienta para registrar y gestionar errores.

## C

- **Code Coverage:** Porcentaje de código cubierto por las pruebas.
- **Compatibility Testing:** Evalúa funcionamiento en distintos dispositivos, navegadores, SOs.
- **Compliance Testing:** Verifica el cumplimiento del sistema con regulaciones legales o estándares de la industria (ej: GDPR, HIPAA).
- **Contract Testing:** Verifica que los contratos entre servicios (por ejemplo, APIs) se cumplan.
- **Continuous Integration (CI):** Integración frecuente del código para detectar errores temprano.
- **Continuous Testing:** Pruebas automatizadas integradas en el ciclo de desarrollo continuo.

## D

- **Data-Driven Testing:** Técnica de automatización donde los casos de prueba se alimentan con datos externos (ej: archivos CSV, Excel).
- **Defect (Defecto):** Imperfección que impide que el sistema funcione como se espera.
- **Defect Density:** Métrica que indica el número de defectos por unidad de medida (ej: LOC, módulos, casos de uso).
- **Defect Life Cycle:** Fases típicas: Nuevo → Asignado → En progreso → Resuelto → Verificado → Cerrado / Reabierto.
- **Definition of Done (DoD):** Criterios que definen cuándo una historia está completamente terminada.
- **Definition of Ready (DoR):** Condiciones mínimas para considerar una historia lista para comenzar.
- **Documentation Testing:** Revisión de documentación técnica y funcional.
- **Dynamic Testing:** Pruebas que requieren la ejecución del sistema.

## E

- **End-to-End Testing:** Pruebas completas desde el inicio hasta el final de un flujo.
- **Entry Criteria:** Condiciones mínimas que deben cumplirse para comenzar una fase de pruebas.
- **Equivalence Partitioning:** Técnica que divide datos en clases válidas/invalidas.
- **Error Guessing:** Técnica basada en experiencia del tester para encontrar errores comunes.
- **Exit Criteria:** Requisitos que deben cumplirse para finalizar formalmente una fase de pruebas.
- **Exploratory Testing:** Exploración libre del sistema por parte del tester.

## F

- **Functional Testing:** Pruebas basadas en los requerimientos funcionales.
- **Fuzz Testing:** Técnica que introduce entradas inválidas para encontrar errores o vulnerabilidades.

## G

- **Grey Box Testing:** Mezcla de pruebas de caja negra y caja blanca.
- **GUI Testing:** Verifica el correcto funcionamiento de la interfaz gráfica de usuario.

## H

- **Happy Path Testing:** Valida el flujo ideal donde todo funciona correctamente, sin errores.
- **High-Level Test Cases:** Casos de prueba generales, centrados en objetivos amplios.

## I

- **Installation Testing:** Evalúa si la app se instala/desinstala correctamente.
- **Integration Testing:** Verifica la interacción entre distintos módulos del sistema.
- **Internationalization Testing:** Verifica si el software puede adaptarse a diferentes idiomas/regiones.
- **ISO/IEC 29119:** Estándar internacional de procesos y documentación para pruebas de software.

## J

- **JUnit:** Framework para pruebas unitarias en Java.

## K

- **Keyword-Driven Testing:** Automatización basada en palabras clave que representan acciones.

## L

- **Load Testing:** Evalúa el rendimiento bajo una carga específica.
- **Localization Testing:** Verifica adaptación cultural, lingüística y de formatos (fechas, moneda, etc.).

## M

- **Manual Testing:** Pruebas realizadas sin automatización.
- **Mocking:** Simulación de componentes o servicios para aislar pruebas.
- **Mutation Testing:** Introduce errores intencionados en el código para verificar la efectividad de los tests.

## N

- **Negative Testing:** Pruebas con entradas incorrectas para validar el manejo de errores.
- **Non-Functional Testing:** Pruebas de atributos no funcionales (rendimiento, seguridad, usabilidad, etc.).

## O

- **Operational Testing:** Evalúa el comportamiento en condiciones reales de operación.
- **OWASP Top 10:** Lista de las 10 vulnerabilidades de seguridad más críticas según la Open Web Application Security Project.

## P

- **Penetration Testing:** Pruebas de seguridad que simulan ataques reales.
- **Performance Testing:** Evalúa velocidad, respuesta y estabilidad bajo carga.
- **Postman:** Herramienta popular para pruebas de APIs REST.

## Q

- **Quality Assurance (QA):** Actividades preventivas enfocadas en asegurar la calidad desde el proceso.
- **Quality Control (QC):** Actividades de verificación centradas en encontrar errores en el producto.

## R

- **Regression Testing:** Verifica que funcionalidades existentes no se rompan por cambios recientes.
- **Retesting:** Pruebas que se hacen después de corregir un defecto para verificar que se haya solucionado.
- **Risk-Based Testing:** Prioriza pruebas según la probabilidad e impacto del fallo.

## S

- **Sanity Testing:** Pruebas rápidas para verificar si el sistema está listo para probarse a fondo.
- **Scalability Testing:** Evalúa si el sistema puede crecer en volumen, usuarios o carga.
- **Security Testing:** Busca vulnerabilidades que puedan comprometer la confidencialidad, integridad o disponibilidad.
- **Selenium:** Conjunto de herramientas para automatizar pruebas web.
- **Shift-Left Testing:** Enfoque que integra pruebas tempranas en el ciclo de desarrollo para detectar errores antes.
- **Smoke Testing:** Conjunto básico de pruebas para validar que el sistema esté estable.
- **Stability Testing:** Evalúa la robustez del sistema en el tiempo, incluso con carga.
- **Static Testing:** Evaluación de documentos, código o diseño sin ejecutar el sistema (ej: revisiones de código).
- **Stress Testing:** Pruebas que exceden los límites operativos normales para evaluar la robustez.
- **System Testing:** Pruebas del sistema completo.

## T

- **Test Case:** Documento o script que define los pasos para validar un comportamiento esperado.
- **Test Coverage:** Porcentaje del sistema cubierto por pruebas, puede ser funcional, de código, de requerimientos, etc.
- **Test Data:** Conjunto de datos utilizado para ejecutar pruebas.
- **Test Driven Development (TDD):** Metodología donde se escriben pruebas antes del código productivo.
- **Test Environment:** Configuración de hardware, software y red donde se ejecutan las pruebas.
- **Test Plan:** Documento que detalla el enfoque, recursos y cronograma para las actividades de prueba.
- **Test Reusability:** Capacidad de reutilizar pruebas existentes para nuevos ciclos, versiones o productos.
- **Test Scenario:** Descripción de alto nivel de una funcionalidad a probar, más amplio que un caso de prueba.
- **Test Strategy:** Alta visión o enfoque organizacional sobre cómo se harán las pruebas en el proyecto/producto.
- **Test Suite:** Conjunto de casos de prueba agrupados por funcionalidad, módulo o propósito.
- **Testing Pyramid:** Representa cómo distribuir los tipos de pruebas para lograr eficiencia y velocidad.

## U

- **Unit Testing:** Pruebas a nivel de función/método/clase.
- **Usability Testing:** Evalúa la facilidad con la que los usuarios interactúan con el sistema.

## V

- **Validation (Validación):** Confirma que el producto cumple con las necesidades del usuario. "¿Estamos construyendo el producto correcto?"
- **Verification (Verificación):** Confirma que el producto se construye según los requisitos. "¿Estamos construyendo el producto correctamente?"
- **Volume Testing:** Evalúa el rendimiento del sistema con grandes cantidades de datos.

## W

- **White Box Testing:** Pruebas que examinan la estructura interna/código del sistema.

## X

- **XSS Testing:** Pruebas específicas para detectar vulnerabilidades de Cross-Site Scripting.
